## Three Review Modes

The skill supports three distinct modes detected from the user's request. **Never guess the mode — ask if ambiguous.**

| Mode | Triggers | Diff command |
|------|----------|--------------|
| **A — GitHub PR** | PR number, GitHub URL, "review this PR" | `gh pr view <N>` + `gh pr diff <N>` |
| **B — Committed range** | Branch name, SHA, "vs staging", "last N commits" | `git diff <base>..<head>` — ask for base if not specified |
| **C — Staged (pre-commit)** | "staged files", "before I commit", "my changes" | `git diff --cached` |

### Why the mode matters

The most common failure in this skill is running the wrong diff command, which causes the reviewer to examine the wrong code entirely. Specifically:

- `git merge-base HEAD origin/main` + `git diff` gives the **committed branch diff** — it has no concept of the index
- `git diff --cached` gives only **staged (indexed) changes** — not committed, not the branch tip
- `gh pr diff` gives the **PR diff as GitHub sees it** — may differ from local if you haven't pushed

### Diff is always inlined

The orchestrating agent collects the diff and inlines it into the subagent prompt via `{FULL_DIFF}`. The subagent is explicitly told NOT to run git commands. This prevents the subagent from recomputing the wrong range.

## Design Decisions

### Meticulous team lead persona
The reviewer is framed as a senior team lead who is the "quality gate between implementation and the codebase." This persona avoids being pedantic while still being thorough. Key traits: cares about code quality, practical not perfectionist, flags real problems not style preferences.

### 8 review lenses (ordered)
The review protocol walks through 8 lenses in order:
1. **Diff Scope** — Enforce that only changed lines are reviewed (CRITICAL per user)
2. **Architecture & Design** — Dependency rules, library types, container pattern
3. **Engineering Principles** — DRY, SOLID, YAGNI, KISS, Separation of Concerns
4. **Library Best Practices** — SWR, Zustand, RHF+Zod, Radix, Next.js 16, Tailwind, React 19
5. **Performance** — Re-renders, memoization, bundle size, data fetching, rendering
6. **HTML Semantics & Accessibility** — Semantic elements, heading hierarchy, ARIA
7. **Correctness & Edge Cases** — Logic errors, type safety, error handling
8. **Naming & Conventions** — Project naming rules

Ordering matters: Diff Scope first ensures the reviewer doesn't waste time on old code. Architecture before library specifics ensures structural issues are caught first.

### Diff-only review policy
Prominently placed as Lens 1 AND in the architecture checklist. Old code problems go in a separate "Pre-existing Issues" section that doesn't count toward the assessment. This matches the project's old code policy from `/docs/`.

### Reference files injected into prompt
Rather than having the subagent read files itself (which costs turns and tokens), the orchestrating agent reads both reference files and injects them into the prompt via `{ARCHITECTURE_CHECKLIST}` and `{LIBRARY_BEST_PRACTICES}` placeholders. This gives the reviewer full context in a single dispatch.

### Architecture decision analysis
The reviewer doesn't just check rules — it independently evaluates architecture decisions. Even "correct" decisions may have simpler alternatives. This addresses the user's request: "analyze the architecture decisions to improve it."

### Severity calibration
Four severity levels plus a separate "Architecture Suggestions" category. Architecture suggestions are framed as alternatives, not defects, which prevents them from inflating the issue count while still providing value.

### Performance checks included per user request
Lens 5 covers: re-renders, memoization, bundle size, data fetching patterns, and rendering performance. These are common issues in React/Next.js apps.

### HTML semantics per user request
Lens 6 covers semantic HTML and accessibility. Checks for proper landmarks, heading hierarchy, interactive elements, form labels, image alt text, table structure, and dialog patterns.

## How to Update

### When architecture rules change
Update `references/architecture-checklist.md`. Re-read the changed `/docs/` files and condense the relevant rules.

### When new libraries are adopted
Add a new section to `references/library-best-practices.md` with bad/good patterns.

### When new review lenses are needed
Add a new numbered lens to `code-reviewer.md` in the Review Protocol section. Keep the ordering logical (structural before specific).

### When the output format needs adjustment
Edit the Output Format section in `code-reviewer.md`. The severity levels and structure should stay stable — change only when there's a clear need.

### When the persona needs adjustment
Edit the opening paragraph of `code-reviewer.md`. The tone calibration (meticulous but practical) was deliberately chosen — avoid making it too strict or too lenient.
